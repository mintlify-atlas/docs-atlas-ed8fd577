Ayush Synapse uses **JWT (HS256)** for authentication. In demo mode, any caller can obtain a token from `POST /auth/login` — no credentials or prior registration is needed. The returned token must then be included in the `Authorization` header of all protected endpoint requests.

---

## POST /auth/login

Generates a signed JWT token for the supplied user identity and role.

| Property | Value |
|----------|-------|
| **Method** | `POST` |
| **Path** | `/auth/login` |
| **Auth required** | No |
| **Content-Type** | `application/json` |

### Request Body

<ParamField body="user_id" type="string" default="demo_user">
  An identifier for the requesting user. Any non-empty string is accepted in demo mode. Echoed back in the response.
</ParamField>

<ParamField body="role" type="string" default="clinician">
  The role to embed in the token. Controls permission checks on protected endpoints.

  Accepted values: `clinician`, `admin`
</ParamField>

### Response Fields

<ResponseField name="message" type="string">
  Always `"Authentication successful"` on a valid request.
</ResponseField>

<ResponseField name="token" type="string">
  The signed JWT string. Pass this value in the `Authorization: Bearer` header of subsequent requests.
</ResponseField>

<ResponseField name="user_id" type="string">
  The `user_id` value echoed from the request body (or `"demo_user"` if omitted).
</ResponseField>

<ResponseField name="role" type="string">
  The role embedded in the token (`"clinician"` or `"admin"`).
</ResponseField>

<ResponseField name="permissions" type="object">
  A set of boolean flags describing what this token authorises.

  <Expandable title="permissions fields">
    <ResponseField name="can_access_all_endpoints" type="boolean">
      Whether the token grants access to all API endpoints.
    </ResponseField>
    <ResponseField name="can_view_all_data" type="boolean">
      Whether the token allows viewing all code and mapping data.
    </ResponseField>
    <ResponseField name="can_perform_translations" type="boolean">
      Whether the token permits calls to `POST /translate`.
    </ResponseField>
    <ResponseField name="can_access_fhir_resources" type="boolean">
      Whether the token permits access to FHIR CodeSystem and ConceptMap endpoints.
    </ResponseField>
  </Expandable>
</ResponseField>

<ResponseField name="demo_mode" type="boolean">
  `true` when the server is running with `DEMO_MODE=True`. Indicates that all features are accessible regardless of role.
</ResponseField>

### Example Request

```bash
curl -X POST http://localhost:5000/auth/login \
  -H "Content-Type: application/json" \
  -d '{"user_id": "dr_sharma", "role": "clinician"}'
```

```json

```

## Using the Token

Include the token from the login response as a `Bearer` credential in the `Authorization` header of every protected request.

```http
Authorization: Bearer <token>
```

<CodeGroup>

```bash curl
curl -X GET http://localhost:5000/namaste \
  -H "Authorization: Bearer eyJ0eXAiOiJKV1Qi..."
```

```python Python (requests)
import requests

token = "eyJ0eXAiOiJKV1Qi..."  # token from /auth/login

headers = {
    "Authorization": f"Bearer {token}",
    "Content-Type": "application/json",
}

response = requests.get("http://localhost:5000/namaste", headers=headers)
print(response.json())
```

```javascript JavaScript (fetch)
const token = "eyJ0eXAiOiJKV1Qi..."; // token from /auth/login

const response = await fetch("http://localhost:5000/namaste", {
  method: "GET",
  headers: {
    Authorization: `Bearer ${token}`,
    "Content-Type": "application/json",
  },
});

const data = await response.json();
console.log(data);
```

</CodeGroup>

## Token Expiry

Tokens are valid for **24 hours** from the time of issue (`iat` claim). After expiry, the API returns:

```json
{
  "error": "Token has expired"
}
```

Re-authenticate by calling `POST /auth/login` again. There is no refresh-token flow; a new full login is required.

---

## Error Responses

| HTTP Status | Condition | Response body |
|-------------|-----------|---------------|
| `401` | `Authorization` header is absent | `{"error": "Authorization header is required"}` |
| `401` | Header format is not `Bearer <token>` | `{"error": "Invalid Authorization header format"}` |
| `401` | Token signature is invalid or malformed | `{"error": "Invalid token"}` |
| `401` | Token has passed its 24-hour expiry | `{"error": "Token has expired"}` |
| `403` | Token is valid but role does not meet endpoint requirement | `{"error": "Access denied. Required role: <role>"}` |

---

<Warning>
  The default `JWT_SECRET_KEY` shipped in `.env.example` (`namaste-icd11-mvp-demo-secret-key-2025`) is public knowledge. **Never use this value in a public or production deployment.** Generate a cryptographically random secret and set it via the `JWT_SECRET_KEY` environment variable before exposing the API to any network.

  ```bash
  # Generate a strong key (Linux / macOS)
  python -c "import secrets; print(secrets.token_hex(32))"
  ```
</Warning>
